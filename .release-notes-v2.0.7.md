## Cove Nexus v2.0.7

Four user-facing additions plus a config-persistence fix from a follow-up review.

### Added

- **Drag-and-drop tool reordering.** Click and drag any card into a new position from any view (All Programs, Installed, Updates, Not Installed, categories, Bookmarks). The order persists across restarts. Hidden tools keep their relative position when you reorder a filtered slice.
- **Bookmarks.** Star button on every card. New "Bookmarks" entry in the sidebar shows only starred tools. Bookmarked tools float to the top in every view. Empty state when nothing is starred.
- **Light/dark theme toggle.** Sun/moon button in the titlebar. Choice persists across restarts. A pre-paint inline script reads the saved theme before the first stylesheet resolves, so light-mode users don't see a dark flash on launch. Every surface (cards, modals, settings, release notes, status pills, tweaks panel) is theme-aware.
- **Update detection hardening.** The version comparator now does a real semver compare after stripping the `v` prefix from both sides — no more false positives or false negatives from prefix mismatches between the local registry and GitHub's tag format. The latest-known tag for each tool is cached in `installs.json`, so a transient GitHub rate-limit no longer hides a previously-detected update. New "update unknown" pill state surfaces the case where GitHub returned no data and there's no cached value, instead of silently letting the user infer "you're on the latest". A `DEBUG_UPDATE_CHECK=1` env var prints a per-tool diagnostic for troubleshooting.

### Fixed

- **Preference round-trip.** Saving one preference no longer silently erases the others on the next write. `readConfig()` now validates and returns the new `toolOrder`, `bookmarks`, and `theme` fields, so an `setPreferences({ theme })` call doesn't drop your bookmarks. Caught in a Codex follow-up review.

### Build & release

- **GitHub Actions release pipeline.** Tagging `v*` now drives the release end-to-end: a Linux runner builds the AppImage and `.deb`; a Windows runner builds `Setup.exe` and `Portable.exe`; a finalize job downloads the auto-update metadata files (`latest*.yml`), generates `.sha256` sidecars for them, and promotes the draft to public. Every shipping asset (binaries, blockmaps, auto-update metadata) gets a `<file>.sha256` sidecar.
